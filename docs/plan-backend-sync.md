# План: синхронизация МП с бэкендом (offline-first)

Статус: реализован (июнь 2026), кроме осознанно отложенного. Бэкенд: `karto4ki_backend`
(Ktor, `/v1/*`). Отклонения от плана при реализации:

- Auth пока тестовый: статический токен через `--dart-define` (`AUTH_TOKEN`, JWT или PAT
  со scope'ами sync) — `TokenAuthService`. Supabase SDK, экран входа, signIn/signOut —
  отложены (п. 8 в силе как целевое состояние).
- На бэке при удалении теста существовал каскадный soft-delete карточек — убран
  (иначе не работает «карточки оживают при restore», п. 4).
- Push-конфликты: к `server_newer`/`deleted` добавлен `reason="not_found"` (restore
  несуществующей сущности), `SyncConflictDto.server` стал nullable.

## Зафиксированные решения

### 1. Архитектура: локальная БД — единственный источник правды для UI

UI **всегда** читает только из Drift. Сеть — фоновый процесс синхронизации, который пишет
в ту же БД. Не «запрос в сеть с fallback на кэш».

- Читающие методы существующих репозиториев не меняются.
- Неавторизованный пользователь = синхронизация выключена, всё работает как сейчас.
- Мутации работают по outbox-паттерну: запись локально + `sync_status = pending`,
  фоновый синк отправляет при первой возможности.

### 2. ID: клиентские UUID

Локальные первичные ключи мигрируют с `INTEGER AUTOINCREMENT` на `TEXT` (UUID v4,
генерируется клиентом при insert). Бэкенд принимает клиентские id через `/sync/push`.
Без этого offline-создание сущностей не синхронизируется. Это первый этап работ.

### 3. Метка синхронизации — только на клиенте

На бэкенде флаг «синхронизировано» не нужен: факт существования записи на сервере и есть
синхронизированность. На клиенте у `tests`, `test_groups`, `cards`:

- `sync_status`: `local` (синк не активен) | `pending` (не отправлено) |
  `pending_restore` (восстановление не отправлено) | `synced`;
- `deleted_at` (nullable) — локальный soft delete.

### 4. Удаление: soft delete с обеих сторон, удалённое = «заморожено», с восстановлением

- На бэке уже есть `deleted_at` + событие `op: "delete"` в `/sync/changes`. Не трогаем.
- Изменения по soft-deleted сущности **игнорируются** (см. п. 10 — доработка push).
- Pulled `delete` на клиенте не стирает строку, а ставит локальный `deleted_at`.
  UI показывает тест как «Удалён»: серый, бейдж, редактирование заблокировано.
  Если были pending-правки — подсветка «правки не будут синхронизированы».
- **Restore — явная операция**, не «воскрешение через upsert» (иначе залежавшаяся
  офлайн-правка молча восстановит удалённый тест). Дёшево, т.к. при удалении теста
  бэк не трогает `deleted_at` карточек (только эмитит delete-события): restore теста =
  `deleted_at = NULL` одной строки, карточки оживают автоматически; индивидуально
  удалённые до этого карточки остаются удалёнными.
  - Бэк: `restoreInTransaction` в репозиториях; `POST /tests/{id}/restore`
    (+ группы) под scope `content:delete`; в `/sync/push` третий op `"restore"`
    (payload не нужен) → снять `deleted_at` + заэмитить **обычные upsert-события**
    по тесту и карточкам (pull-протокол не меняется); связи `groupIds` фильтруются
    на удалённые группы.
  - Клиент: кнопка «Восстановить» на состоянии «Удалён»; локально `deleted_at = NULL`
    и статус `pending_restore`; в pull — `upsert` по локально-удалённой записи снимает
    локальный `deleted_at`.
  - Статистика привязана к `question_key` ⇒ после restore история ответов на месте.
- Принудительное (физическое) удаление — отдельная задача позже; состояние «удалён»
  на клиенте — готовый фундамент под неё («корзина» → purge). Restore после purge
  невозможен (404) — обработать при реализации purge.

### 5. Цикл синхронизации: pull → reconcile → push

1. **Pull**: `GET /sync/changes?since=<cursor>` циклом, пока `hasMore`; курсор — в настройках.
   - `delete` применяем всегда, включая записи с локальным `pending` (удаление побеждает правку);
   - `upsert` поверх локальной `pending`-записи **не** перезаписываем — локальная версия
     уйдёт в push;
   - остальные `upsert` пишем в Drift со статусом `synced`.
2. **Reconcile**: отбрасываем pending-правки сущностей, чьи родители удалены.
3. **Push**: все `pending`/`pending_restore` в порядке зависимостей (groups → tests → cards),
   батчами до 1000; `upsert` с полным payload, `delete` для строк с `deleted_at`,
   `restore` для `pending_restore`. Успех → `synced`.

### 6. Конфликты: молча принимаем сервер

Бэкенд решает по LWW (`updatedAt`, серверное время при apply ⇒ фактически «кто первый
запушил — тот победил»). Принято допущение: один пользователь не редактирует одну сущность
с двух устройств одновременно. Поэтому на `server_newer` клиент молча применяет серверную
копию из ответа, снимает `pending`, пишет в лог. Никакого merge-UI и уведомлений.

### 7. Запуск синка: автоматический, кнопка — вспомогательная

- Авто-триггеры: старт приложения, восстановление сети (`connectivity_plus`),
  debounce после локальной мутации.
- Ручной запуск: pull-to-refresh на списках.
- В настройках — строка статуса («Синхронизировано N мин назад / M ждут отправки»)
  с кнопкой «Синхронизировать сейчас».
- Отдельной обязательной кнопки как основного механизма нет.

### 8. Auth: Supabase JWT

- `supabase_flutter`; бэкенд верифицирует JWT (`sub` = userId), запись в `users`
  создаётся автоматически при первом запросе (JIT-provisioning) — отдельной регистрации нет.
- Токены в secure storage, refresh делает SDK; auth-interceptor в dio (Bearer, ретрай на 401).
- Без авторизации: всё локально, `sync_status = local`, sync-слой неактивен.
- Первый вход: все локальные данные помечаются `pending` и выгружаются на сервер.
- Выход: данные остаются локально, синк выключается.
- Вход под другим аккаунтом: предупреждение + очистка локальной БД (БД однопользовательская).

### 9. Статистика: отправляем события, агрегаты принимаем с сервера

`/sync/push` статистику игнорирует (pull-only). Поэтому:

- Новая локальная таблица `answer_events` (outbox: `card_id`, `is_correct`, `answered_at`,
  `sent`); пишется при сохранении результатов сессии.
- Отправка: `POST /v1/stats/answers/batch`; успешные помечаются `sent`.
- Агрегаты `question_stats` приходят через pull (`entity: stats`) и принимаются как
  авторитетные (включают события всех источников, в т.ч. MCP).
- Известное следствие: правка текста карточки меняет `question_key` ⇒ статистика по старой
  формулировке остаётся «осиротевшей», по новой копится с нуля. Это текущее поведение
  и локально, не баг синка.

### 10. MCP: только чтение, добавление, обновление и статистика — без удаления

Сейчас scope'ы PAT проверяются только на `/sync` и `/import`; CRUD доступен любому PAT.
Доработки бэка:

- Новый scope `content:delete`; `requireScope` на `DELETE /tests/{id}`, `DELETE /groups/{id}`,
  `DELETE /cards/{id}`, `POST /tests/{id}/restore` и **`POST /tests/{id}/merge`**
  (merge soft-delete'ит исходный тест). JWT проходит всегда; MCP-токенам scope не выдаётся.
- MCP-токенам не выдаётся `sync:write` (через `/sync/push` можно протащить `op: delete`;
  push MCP не нужен — он работает через CRUD).
- Из `mcp/server.py` убрать тулзы `delete_test`, `delete_group`, `delete_card`,
  `merge_tests`, `delete_me` (UX-слой; защита — scope'ы).

## Этапы работ

### МП (karto4ki)

| Этап | Содержание | Ключевые файлы |
|---|---|---|
| 0. Миграция схемы (v9) | PK → TEXT UUID в `tests`, `test_groups`, `cards` (+ перепривязка FK); колонки `sync_status`, `deleted_at`; таблица `answer_events`; DAO-запросы дополняются `WHERE deleted_at IS NULL` | `lib/persistence/database/app_database.dart`, `*.drift` |
| 1. Auth | `supabase_flutter`, экран входа, secure storage, состояние авторизации в `AppScope` | `lib/app/di/` |
| 2. Сетевой слой | `dio` + `retrofit`: `GroupsApi`, `TestsApi`, `CardsApi`, `StatsApi`, `SyncApi`; DTO 1:1 с бэком; auth-interceptor; маппинг ответов `\|`-строка ↔ `answers: []` через `AnswerParser` | `lib/core/network/` (новый), скилл `create-api` |
| 3. SyncManager | Цикл pull → reconcile → push (п. 5); курсор в настройках; конфликты по п. 6; мутации в существующих репозиториях ставят `pending`, delete → soft delete | `lib/core/sync/` (новый), репозитории фич |
| 4. UI | Бейдж «не синхронизировано» (`pending`), состояние «Удалён» + кнопка «Восстановить» (п. 4), статус-строка и триггеры синка (п. 7) | entity фич, экраны списков, настройки |
| 5. Статистика | Outbox `answer_events` → batch-отправка; приём агрегатов через pull | `lib/feature/question_stats/` |

Порядок: 0 → 1 → 2 → 3 → 4 → 5. Этапы 0 и 3 — самые трудоёмкие; 1–2 механические.

### Бэкенд (karto4ki_backend)

| Этап | Содержание |
|---|---|
| B1. Push: deleted = skip | В `applyTest`/`applyCard`/`applyGroup` upsert по soft-deleted сущности (и карточке удалённого теста) → per-change результат `reason: "deleted"` (или skip), а не validation-ошибка на весь батч |
| B2. Scope на удаление | `content:delete` + `requireScope` на DELETE-роутах, restore и merge (п. 10) |
| B3. MCP-тулзы | Убрать delete/merge-тулзы из `mcp/server.py` |
| B4. Restore | `restoreInTransaction` в репозиториях; `POST /tests/{id}/restore` (+ группы); op `"restore"` в `/sync/push` с эмиссией upsert-событий (п. 4) |

B1 — обязателен до запуска клиентского синка (иначе одна правка по удалённому тесту
блокирует весь push устройства). B2–B4 независимы.

## Вне скоупа (осознанно отложено)

- Принудительное физическое удаление (purge / «корзина»).
- Merge-UI для конфликтов.
- Полная матрица scope'ов (`content:read`/`content:write`/`stats:write`).
- Мультиаккаунтность на одном устройстве.
